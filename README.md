# Provenance Guard

A content-provenance API that estimates whether a piece of text was written by a human or generated by an AI tool, attaches an honest, hedged transparency label to the result, and lets creators appeal a classification. Built with Flask + SQLite, using a three-signal ensemble (one LLM signal, two pure-Python heuristic signals).

The core design stance: **this is a probabilistic judgment, not a verdict.** A false positive — calling a human's work AI-generated — is the costly, reputation-damaging error, so the system is built to say "I don't know" loudly and often rather than overclaim.

**Quick start**

python -m venv .venv
.venv\Scripts\activate            # Windows
pip install -r requirements.txt

# Signal 1 calls Groq, so a key is required:
echo GROQ_API_KEY=your_key_here > .env

**1. Architecture overview — the path a submission takes**

```
POST /submit  { text, creator_id }
        │
        ▼
   input validation ───► 400 if text/creator_id missing or empty
        │
        ▼
  ┌─────────────────────────────────────────────────────────┐
  │  THREE DETECTION SIGNALS  (signals.py)                    │
  │                                                           │
  │  Signal 1  get_llm_score()          → llm_score      [0,1]│  (Groq, network call)
  │  Signal 2  get_stylometric_score()  → stylo_score    [0,1]│  (pure Python)
  │  Signal 3  get_linguistic_patterns()→ ling_score     [0,1]│  (pure Python, stretch)
  └─────────────────────────────────────────────────────────┘
        │
        ▼
   combine_signals()  →  confidence  (ensemble mean, [0,1] = P(AI))
        │
        ▼
   attribution_from_score()   →  "likely_human" | "uncertain" | "likely_ai"
   get_transparency_label()   →  one of three exact human-readable strings
        │
        ▼
  ┌─────────────────────────────────────────────────────────┐
  │  PERSISTENCE  (store.py, SQLite)                          │
  │   content     — canonical, MUTABLE record (status flips   │
  │                 classified → under_review on appeal)      │
  │   audit_log   — APPEND-ONLY snapshot of every decision    │
  └─────────────────────────────────────────────────────────┘
        │
        ▼
   JSON response  { content_id, attribution, confidence, label, signals, status }
```

If Signal 1's network call fails the request returns `502` with an honest error rather than silently degrading to the heuristic signals — a partial classifier shouldn't masquerade as a full one.

The audit log is deliberately separate from the content table: an appeal can change the *live* status of a piece of content without ever rewriting *history*. Every audit entry is a self-contained JSON snapshot (creator, every signal score, the LLM's own verdict and one-line reasoning, the exact label shown), so a reviewer can reconstruct a decision without re-running detection.

**Files:** `app.py` (routes, ensemble orchestration, rate limiter, dashboard) · `signals.py` (the three signals, scoring, thresholds, label text) · `store.py` (SQLite, audit trail, verification).

**2. Detection signals — what each measures, why I chose it, what it misses**

I use three signals on the principle that no single lens is trustworthy on its own. They're chosen to be **independent**: they fail in different ways, so when they agree, I can trust the result more, and when they disagree, the score drifts toward the uncertain band on its own.

**Signal 1 — LLM classification (Groq, `llama-3.3-70b-versatile`)**

- **Measures:** Holistic, *semantic* AI-ness. The model is prompted as a forensic analyst and returns `{ai_probability, verdict, reasoning}`. `ai_probability` becomes `llm_score`. Temperature is 0 for the most deterministic output a classifier can get.
- **Why I chose it:** It's the only signal that reads *meaning*. It catches the things heuristics can't articulate — generic phrasing, "lived-in" specificity vs. boilerplate, the particular even cadence of model output. It's the strongest single signal, which is why it had the heaviest weight in the original two-signal design.
- **What it misses:** It's pattern-matching, not ground truth. It can be fooled by lightly-edited AI text, and — the dangerous failure — it can flag careful, formal, or non-native-English human writing as AI because that prose *resembles* model output. It's also non-deterministic across model updates and adds network latency and a hard dependency on an external API.

**Signal 2 — Stylometric heuristics (pure Python)**

- **Measures:** *Structure*, not meaning. Average of three normalized metrics:
  - **Sentence-length variance** (coefficient of variation) — AI tends to produce uniform sentence lengths; humans are bursty.
  - **Type-token ratio** — vocabulary diversity; AI often repeats a narrower vocabulary.
  - **Punctuation density and variety** — humans use punctuation more irregularly.
- **Why I chose it:** It's transparent, deterministic, free, and completely independent of the LLM — it can't be fooled by the same things the LLM is fooled by. It's a structural cross-check.
- **What it misses:** It's statistically meaningless on short text with one or two sentences; there isn't enough data for variance or TTR to mean anything, so it returns a neutral 0.5. It also has no concept of meaning, so polished human prose with even sentences scores AI-like.

**Signal 3 — Linguistic patterns (pure Python, stretch/ensemble)**

- **Measures:** Surface lexical tells. Density of hedging/formal connectors ("however," "moreover," "furthermore," "it is important to note") and *absence* of contractions (AI under-uses contractions; humans use them freely).
- **Why I chose it:** It targets a very specific, recognizable AI register that the stylometric signal doesn't capture and that the LLM signal sometimes under-weights. Cheap to compute and easy to explain.
- **What it misses:** It's the most gameable signal — adding a few contractions or stripping connectors moves it directly. It also penalizes formal-register human writing (academic, legal, technical) that legitimately uses those connectors, and it returns a neutral 0.5 below 10 words.

**The shared blind spot:** all three signals key off *style*, so genuinely human writing in a formal/templated register triggers all three at once — see Known Limitations.

**3. Confidence scoring**

**How signals are combined**

confidence = (llm_score + stylometric_score + linguistic_score) / 3   # ensemble mean

A simple equal-weight average of all three signals, producing `confidence` ∈ [0, 1] = the estimated probability that the text is AI-generated. Each signal gets one vote.

`confidence` maps to an attribution and a label through fixed thresholds (`signals.py`):

| Score range | Attribution | Confidence in the determination |
|---|---|---|
| `0.00 – 0.39` | `likely_human` | High |
| `0.40 – 0.69` | `uncertain` | **Low — system says "I don't know"** |
| `0.70 – 1.00` | `likely_ai` | High |

**Why this approach?** I started with a weighted scheme (`0.6·llm + 0.4·stylometric`) that trusted the LLM most, and kept that as a documented two-signal fallback in `combine_signals()`. But once Signal 3 made it a three-signal system, equal weighting became the more defensible default: averaging independent signals is exactly what pulls a confident-but-wrong single signal back toward the middle when the others disagree. That self-correction is the whole point. A lone LLM saying 0.9 while two heuristics say 0.4 lands at ~0.57 — *uncertain* — which is the honest answer when signals conflict.

**Why is the uncertain band wide (0.40–0.69, ~30% of the range).** This is asymmetric-cost design. Falsely accusing a human of using AI is far more damaging than declining to judge, so I made "I don't know" easy to reach. The threshold for `likely_ai` (0.70) is deliberately higher than the symmetric 0.5 would be.

**Validation — does the score produce meaningful variation?**

This is behavioral, not statistical, calibration: I ran a range of inputs and confirmed the score *orders* them the way a human would and *spreads* them across all three bands rather than clustering. The scores below are real records from the project's SQLite store (`provenance.db`) and the Milestone-4 ensemble tests.

**Example A — High-confidence case (clearly human)**

*"Hey, I'm really happy about this! It's so cool, don't you think?"* — `creator_id: casual-user`

| Signal | Score |
|---|---|
| LLM | 0.40 |
| Stylometric | 0.10 |
| **Confidence (ensemble)** | **0.167** |
| **Attribution** | **`likely_human`** |

Contractions, exclamation, irregular rhythm, casual voice — the heuristics drive this firmly into the human band. The system is *confident*.

**Example B — Lower-confidence case (genuinely ambiguous)**

 *"The implementation demonstrates various analytical capabilities."* — `creator_id: technical-user`

| Signal | Score |
|---|---|
| LLM | 0.80 (verdict: `likely_ai`, "generic phrasing and lacks specific details") |
| Stylometric | 0.483 |
| Linguistic | 0.50 |
| **Confidence (ensemble)** | **0.594** |
| **Attribution** | **`uncertain`** |

The LLM is fairly sure it's AI, but it's a single short, formal sentence — too little signal for the heuristics, which sit near neutral. The disagreement is exactly what *should* produce hedging, and the ensemble lands at 0.594 → **uncertain**. The system declines to accuse.

**The gap between A and B is 0.167 vs 0.594 — a 0.43 spread across two different label variants**, demonstrating the score is not constant.

**Example C — High-confidence AI (Milestone-4 ensemble test)**

High-hedging, connector-heavy AI text scored `(LLM 0.8, stylometric 0.633, linguistic 1.0) → confidence 0.811 → likely_ai`, confirming the top band is reachable. Together, A/B/C land in all three bands.

**What I'd change deploying this for real**

- **Calibrate against a labeled corpus.** "Behavioral calibration" is fine for a class project; production needs real human/AI-labeled data so the 0.70 threshold reflects a measured false-positive rate, not intuition.
- **Length-aware weighting.** The heuristics are noise below ~3 sentences. I'd down-weight or drop them on short text instead of letting their neutral 0.5 dilute a confident LLM read (which is what muddies Example B).
- **Confidence ≠ probability.** I'd report a calibrated interval, not a single point, and surface signal disagreement directly to the user.

**4. Transparency label — the three variants**

The label is plain prose, never a bare verdict, and is intentionally hedged. Below is the **exact text** each variant displays (from `get_transparency_label()` in `signals.py`).

**Variant 1 — High-confidence AI (`confidence ≥ 0.70`)**

This content shows strong indicators of AI generation. Our system is fairly confident this was created or substantially produced by an AI tool.

Even at high confidence, it says "fairly confident" and "indicators," never "this is AI." It avoids sounding accusatory.

**Variant 2 — Uncertain (`0.40 ≤ confidence < 0.70`)**

We can't confidently determine whether this content was written by a human or generated by AI. Treat this result as inconclusive.

A genuine non-answer. No number is foregrounded; no implication of either outcome.

**Variant 3 — High-confidence human (`confidence < 0.40`)**

This content shows strong indicators of human authorship. Our system did not detect significant signs of AI generation.

Phrased as "did not detect signs of AI" rather than "this is human" — an absence of evidence, not proof.

Each `/submit` response also returns `attribution`, `confidence`, and the full per-signal breakdown alongside the label, and every label shown is captured verbatim in the audit log.

**5. Rate limiting**

Applied to `/submit` (the only expensive, abusable endpoint — it makes a paid LLM call per request) via `flask-limiter`, keyed on client IP:

10 per minute;100 per day


**Reasoning for these specific values:**

- **10/minute** — comfortably above a human pasting documents one at a time, but it caps automated scraping at a rate where the per-request Groq cost and latency stay bounded. A real person never needs an 11th classification in the same minute.
- **100/day** — a sustained-abuse ceiling. It stops someone from quietly grinding through thousands of classifications over hours (cost control, and it limits using the endpoint as a free oracle to *probe* the detector and reverse-engineer evasions), while leaving a legitimate heavy user ample headroom.
- **Per-IP, in-memory** for this build. In production I'd move the limiter store to Redis (so limits hold across multiple app instances) and key on authenticated account rather than IP, since IP is both spoofable and shared behind NAT.

Exceeding either limit returns HTTP **429 Too Many Requests**. Read endpoints (`/log`, `/health`, `/dashboard`) are intentionally unlimited.

**6. Known limitations**

**The system will reliably misclassify formal, structured *human* writing — academic abstracts, legal/policy prose, technical documentation, and especially competent non-native-English writing — as AI.** Example B above is a live instance: a real human sentence that the LLM signal rated 0.80 AI.

This isn't a "needs more data" hand-wave; it's a direct consequence of *what my signals measure*. All three key off **style, not provenance**:
- Signal 1 (LLM) learned "AI-like" from text that itself reads even, hedged, and generic — the same register careful or non-native human writers naturally produce.
- Signal 2 (stylometric) rewards uniform sentence length and lower vocabulary diversity — properties of disciplined formal writing.
- Signal 3 (linguistic) explicitly counts "however/moreover/furthermore" and missing contractions — the exact markers of formal academic register.

So formal human prose trips all three at once, and because they fail *in the same direction* the ensemble averaging that protects against single-signal error provides no protection here — agreement raises confidence even though all three are wrong for the same reason. The wide uncertain band and the appeal path are the mitigations, not a fix.

Secondary limitations: heuristics are unreliable below ~3 sentences (return neutral 0.5), and the LLM signal is gameable by light editing.

**7. Spec reflection**

**How the spec (planning.md) helped.** Writing the transparency-label wording and the threshold table *before* coding was the highest-leverage thing I did. Because the three exact label strings and the 0.40 / 0.70 cutoffs were fixed in the plan, `signals.py` had a single source of truth — `get_transparency_label()` and `attribution_from_score()` share the same `HUMAN_CEILING`/`AI_FLOOR` constants — and I never had thresholds drifting between the labeling code and the response code. The plan's "asymmetric cost → wide uncertain band" rationale also kept me from caving to the temptation to tighten the band for cleaner-looking demos.

**Where the implementation diverged, and why.** The spec specified a *two-signal weighted* score: `0.6·llm + 0.4·stylometric`. The shipped default is a *three-signal equal-weight ensemble*. I diverged because I added Signal 3 (linguistic patterns) as a stretch feature, and at that point fixed 0.6/0.4 weights no longer made sense — equal voting is the more honest combiner for three independent signals and gives the self-correcting "disagreement → uncertainty" behavior I wanted. I kept the original 0.6/0.4 formula in `combine_signals()` as a documented fallback for when only two signals are available, so the spec's design is still present, just no longer the primary path.

**8. AI usage**

I used an AI coding assistant throughout, directing it against the plan and verifying every output. Specific instances:

**1. Stylometric signal (Signal 2).** I directed it to write `get_stylometric_score()` implementing the three metrics named in the plan (sentence-length variance, type-token ratio, punctuation density/variety), each normalized to [0,1]. It produced a working function, but its first version normalized raw sentence-length variance directly — which is scale-dependent, so long texts looked artificially AI-like regardless of style. **I overrode this**, switching the metric to coefficient of variation (variance normalized by mean length) so the score is scale-independent, and I tuned the human/AI cutoff constants (CV ≈ 0.3 vs 0.6) by hand against sample texts.

**2. Ensemble combiner + threshold sharing.** I asked it to extend `combine_signals()` for three signals and to generate the label function. It initially wrote a *second*, independent set of threshold constants inside `get_transparency_label()`, duplicating the 0.40/0.70 cutoffs that already lived in `attribution_from_score()`. **I revised this** to make both functions read the same `HUMAN_CEILING`/`AI_FLOOR` module constants, eliminating the risk of the label and the attribution disagreeing. I also rejected its suggestion to default the empty-signal case to 0.0, changing short-text/low-data cases to return a neutral **0.5** instead, consistent with the "don't guess" principle.

**3. Analytics dashboard (stretch).** I directed it to build the `/dashboard` Chart.js view. It produced the HTML/JS, but I caught — during review, not from the AI — that the dashboard reads `content.linguistic_score`, while the AI-generated `save_content()` had silently dropped that column from its INSERT, so Signal 3 was stored as `NULL` and the dashboard's linguistic stats rendered empty. I traced it to the column/value mismatch and fixed `save_content()` to persist `linguistic_score`.

**Rate-limit behavior**
The 11th `/submit` within one minute (or 101st in a day) returns:
```
HTTP/1.1 429 TOO MANY REQUESTS
{ "error": "429 Too Many Requests: 10 per 1 minute" }
```

**Appeal handling (`POST /appeal`)**
Request:
```json
{ "content_id": "2ead8a15-3239-4f7c-bf4d-ca276bef33f2",
  "creator_reasoning": "I wrote this abstract myself; the formal tone is my normal academic style." }
```
Response:
```json
{ "content_id": "2ead8a15-3239-4f7c-bf4d-ca276bef33f2",
  "status": "under_review",
  "message": "Appeal received. A human reviewer will examine your submission." }
```
The content row flips `classified → under_review`; a new append-only `appeal` audit entry is written carrying the creator's reasoning plus the original attribution, confidence, and every signal score — enough context for a manual reviewer to decide without re-running detection. Automated reclassification is intentionally out of scope.
